الهدف

يجب على الوكيل الذكي عند بناء التطبيق أو تحديثه أن يلتزم بإنتاج تطبيق Flutter احترافي، آمن، سريع، قابل للصيانة، قليل استهلاك الموارد، ومواكب لمعايير التطبيقات الحديثة في 2026، مع المحافظة على جودة الكود وتجربة المستخدم وعدم إدخال تغييرات عشوائية أو مكسّرة للنظام.

1) قواعد عامة إلزامية
لا تنفّذ أي تعديل عشوائي أو واسع بدون فهم بنية المشروع الحالية.
افحص الملفات المرتبطة بالمهمة قبل التعديل، ولا تعدّل ملفات غير مرتبطة إلا عند الحاجة الحقيقية.
أي تعديل يجب أن يكون متوافقًا مع البنية الحالية للمشروع.
أي كود جديد يجب أن يكون قابلًا للتوسع وليس مجرد حل مؤقت.
لا تكرّر المنطق في أكثر من مكان، وفضّل إعادة الاستخدام.
لا تكسر التوافق مع الشاشات أو الخدمات أو الـ APIs الحالية إلا إذا كان ذلك مطلوبًا بوضوح.
أي تعديل يجب أن يحافظ على استقرار التطبيق في وضع:
debug
release
Android
iOS إن كان المشروع يدعمه
لا تضع أسرارًا أو مفاتيح حساسة داخل الكود مباشرة.
لا تستخدم حلول hacky أو quick fixes إذا كان بالإمكان بناء حل صحيح ومنظّم.
2) فهم المهمة قبل التنفيذ

قبل البدء بأي تنفيذ، يجب على الوكيل أن:

يحدّد الهدف الفعلي من التعديل.
يحدّد الملفات المتأثرة مباشرة وغير مباشرة.
يراجع الـ widgets / controllers / services / models / repositories المرتبطة.
يراجع المسارات والـ navigation والتبعيات المتأثرة.
يتأكد هل التعديل:
تصميم فقط
منطق أعمال
API integration
state management
performance
bug fix
refactor
يحافظ على السلوك الحالي ما لم يكن المطلوب تغييره صراحة.
3) معايير البنية المعمارية Architecture

يجب الالتزام ببنية واضحة ومنظمة، ويفضل اعتماد فصل مشابه لما يلي:

lib/
  core/
    constants/
    theme/
    utils/
    services/
    network/
    errors/
    widgets/
  features/
    feature_name/
      data/
        models/
        datasources/
        repositories/
      domain/
      presentation/
        screens/
        widgets/
        controllers/
قواعد معمارية
فصل الواجهة عن منطق الأعمال وعن طبقة البيانات.
عدم وضع منطق API داخل الشاشة مباشرة.
عدم وضع parsing معقّد داخل build().
عدم جعل الشاشة العملاقة تحتوي كل شيء.
أي feature جديدة يجب أن تكون معزولة قدر الإمكان.
استخدام Repository pattern أو ما يعادله عند التعامل مع البيانات.
استخدام Service واضح للشبكة والتخزين والإشعارات.
كل Model يجب أن يكون منظمًا وواضحًا وقابلًا للتحويل من/إلى JSON بأمان.
4) إدارة الحالة State Management
يجب استخدام أسلوب إدارة حالة واضح ومتسق على مستوى المشروع.
إذا كان المشروع يعتمد Provider أو Riverpod أو Bloc، فيجب الالتزام بما هو مستخدم حاليًا وعدم إدخال أسلوب آخر إلا لسبب قوي.
يمنع التوسع العشوائي في setState داخل الشاشات الكبيرة.
يجب تمثيل الحالات الأساسية بوضوح:
idle
loading
success
empty
error
refreshing
paginating
عدم خلط حالة الواجهة المؤقتة مع حالة البيانات العامة.
أي state جديدة يجب أن تكون قابلة للاختبار وسهلة التتبع.
5) معايير التصميم UI/UX

عند بناء أو تحديث أي شاشة، يجب أن تكون النتيجة:

حديثة
نظيفة
مريحة بصريًا
متناسقة
مناسبة للجوالات الحديثة في 2026
قواعد واجهة إلزامية
احترام الهوية البصرية للمشروع.
الاعتماد على Design System ثابت:
الألوان
الخطوط
المسافات
الزوايا
الظلال
الأزرار
الحقول
توحيد شكل:
AppBars
Buttons
Inputs
Cards
Empty states
Error states
Loaders
دعم الوضع الداكن إذا كان المشروع يدعمه.
دعم RTL بشكل ممتاز إذا كانت اللغة العربية مستخدمة.
عدم تكديس العناصر أو تكبير عدد المكونات في الشاشة بلا داع.
الحفاظ على وضوح التسلسل البصري hierarchy.
جعل التفاعل واضحًا: ما هو الأساسي، ما هو الثانوي، وما هي الحالة الحالية.
تجربة المستخدم
تقليل عدد النقرات لإنجاز المهمة.
عرض رسائل خطأ واضحة ومفهومة.
توفير empty states مفيدة.
استخدام skeleton/loading مناسب بدل الدوران المزعج الطويل.
جعل النماذج سهلة وواضحة.
دعم حالات الشبكة الضعيفة أو البطء.
توفير feedback بصري عند النجاح أو الفشل.
احترام أحجام الشاشات المختلفة.
عدم إزعاج المستخدم بتنقلات مفاجئة أو رسائل زائدة.
6) الأداء Performance

أي كود جديد يجب أن يُكتب مع مراعاة الأداء من البداية.

قواعد أداء إلزامية
استخدام const حيثما أمكن.
تقليل إعادة البناء غير الضرورية.
تقسيم الشاشات الكبيرة إلى Widgets صغيرة قابلة لإعادة الاستخدام.
عدم وضع عمليات ثقيلة داخل build().
استخدام:
ListView.builder
GridView.builder
lazy loading
عدم تحميل جميع البيانات دفعة واحدة إذا كانت كثيرة.
عدم تحميل صور كبيرة بلا ضغط أو resize مناسب.
تأجيل تحميل الأجزاء غير الضرورية.
تقليل عمق الشجرة في الواجهة عندما يكون ذلك ممكنًا.
عدم تنفيذ عمليات parsing أو filtering الثقيلة على الـ UI thread إذا كانت كبيرة.
تحسين startup time وعدم إضافة تهيئات ثقيلة عند فتح التطبيق.
الصور والملفات
ضغط الصور قبل الرفع.
استخدام صيغ مناسبة مثل WebP عند الإمكان.
عرض صور مصغرة عند الحاجة بدل الأصل الكامل.
عدم إعادة تحميل نفس الصورة أو الملف بلا داع.
7) تقليل استهلاك السيرفر والشبكة

الوكيل الذكي يجب أن يكتب التطبيق بطريقة اقتصادية على مستوى الـ backend والباندويدث.

قواعد إلزامية
استخدام pagination في القوائم الكبيرة.
عدم جلب بيانات لا تحتاجها الشاشة.
عدم تكرار نفس الطلبات عند كل إعادة بناء.
استخدام caching محلي عند الحاجة.
استخدام debounce في البحث.
تجنب polling غير الضروري.
احترام timeouts.
التعامل الذكي مع retries.
استخدام refresh عند الحاجة فقط.
عدم إعادة تحميل كامل البيانات إذا كان بالإمكان تحديث جزء صغير فقط.
دعم offline-first أو last-known-state حيث يناسب.
تقليل عدد request chains المتتابعة قدر الإمكان.
تجميع الطلبات أو تبسيطها عندما يكون ذلك مناسبًا.
API Integration
جميع الطلبات يجب أن تمر عبر طبقة شبكة موحّدة.
التعامل مع الأخطاء يجب أن يكون موحدًا.
دعم token refresh إذا كان النظام يستخدم auth tokens.
عدم افتراض أن كل استجابة ناجحة أو أن كل الحقول موجودة.
كتابة parsing آمن مع fallback مناسب.
تسجيل الأخطاء المهمة للمساعدة في التشخيص.
8) الأمان Security

أي تعديل يجب أن يحافظ على أمان التطبيق والبيانات.

قواعد إلزامية
استخدام HTTPS فقط.
عدم تخزين أسرار داخل الكود.
استخدام secure storage للبيانات الحساسة.
التحقق من إدخالات المستخدم.
منع أي اعتماد على قيم غير موثوقة من العميل.
احترام الصلاحيات وعدم كشف خصائص أو إجراءات لمستخدم غير مصرح له.
التعامل الآمن مع JWT/OAuth2 أو أي نظام توثيق مستخدم.
عدم تخزين tokens في أماكن غير آمنة.
عدم إظهار رسائل أخطاء حساسة للمستخدم النهائي.
تقليل البيانات الحساسة المخزنة محليًا.
عند رفع الملفات، يجب مراعاة:
نوع الملف
الحجم
التحقق
الرسائل الآمنة
9) إدارة الأخطاء Error Handling
لا تترك exceptions صامتة.
أي فشل يجب أن يعالج بطريقة واضحة.
عرض رسالة مناسبة للمستخدم.
تسجيل الخطأ فنيًا إن لزم.
التفريق بين:
network error
validation error
unauthorized
server error
parsing error
unknown error
لا تستخدم catch عام بدون سبب وبلا logging.
لا تجعل التطبيق يتوقف بسبب null أو parsing بسيط.
10) قابلية الصيانة Maintainability
كتابة كود واضح ومنظم.
اختيار أسماء مفهومة للملفات والكلاسات والدوال والمتغيرات.
تجنب الملفات العملاقة.
تقليل التكرار.
استخراج المكونات المشتركة.
إضافة تعليقات فقط عند الحاجة للتوضيح المنطقي، وليس لشرح البديهيات.
عدم ترك كود ميت أو تجريبي أو مطبوعات print غير ضرورية.
استخدام logging منظم بدل العشوائية.
الحفاظ على تنسيق موحد للكود.
احترام lint rules الخاصة بالمشروع.
11) التوافق مع المشروع الحالي
لا تغيّر naming conventions الحالية إلا لسبب معماري واضح.
لا تدخل مكتبات جديدة إلا إذا كانت:
ضرورية
مستقرة
مدعومة
شائعة
تضيف قيمة واضحة
راعِ توافق التعديل مع:
navigation
theme
dependency injection
state management
localization
analytics
existing APIs
لا تكسر العقود الحالية للـ services أو repositories بدون تحديث جميع الاستخدامات المرتبطة.
12) الاختبارات Testing

أي ميزة جديدة أو تعديل مهم يجب أن يراعي الاختبارات.

الحد الأدنى
إضافة أو تحديث unit tests عند وجود منطق أعمال.
إضافة أو تحديث widget tests للمكونات المهمة.
التأكد من عدم كسر السيناريوهات الأساسية.
عند إصلاح bug، يفضّل إضافة test يمنع تكراره.
ما يجب التحقق منه
loading states
empty states
error states
parsing
validation
state transitions
navigation المهمة
التوافق مع أكثر من حجم شاشة إن لزم
13) التوثيق Documentation

عند كل تعديل مهم، يجب أن تكون النتيجة مفهومة للمطور التالي.

يجب توثيق:
سبب التعديل
الملفات المعدلة
أي dependency جديدة
أي متغيرات بيئية جديدة
أي endpoint جديد
أي breaking change
خطوات التشغيل أو الاختبار إن لزم
14) قواعد خاصة بالشاشات والـ Widgets
كل Widget يجب أن يكون له دور واضح.
تجنب إنشاء Widgets ضخمة جدًا.
يفضّل استخراج الأجزاء المتكررة إلى Widgets مستقلة.
لا تجعل الشاشة مسؤولة عن كل شيء.
لا تخلط بين rendering وbusiness logic.
أي عنصر قابل لإعادة الاستخدام يجب نقله إلى shared/core widgets إذا كان عامًا.
15) قواعد خاصة بالنماذج Forms
التحقق من الإدخال يجب أن يكون واضحًا.
استخدام validation مناسب.
إظهار الرسائل بجانب الحقول أو بشكل مفهوم.
منع الإرسال المتكرر أثناء التحميل.
دعم الحالات:
success
invalid fields
server error
timeout
تحسين تجربة لوحة المفاتيح والتنقل بين الحقول.
احترام accessibility في الحقول والأزرار.
16) قواعد خاصة بالتنقل Navigation
يجب أن يكون التنقل واضحًا ومستقرًا.
عدم التسبب في stack عشوائي.
دعم الرجوع بشكل منطقي.
تمرير البيانات بين الشاشات بطريقة آمنة.
حماية الشاشات التي تتطلب auth.
عدم فتح شاشات حساسة بدون تحقق من الحالة أو الصلاحية.
17) قواعد خاصة بالمكتبات الخارجية

قبل إضافة أي package:

تأكد أنها نشطة ومدعومة.
راجع تأثيرها على الحجم والأداء.
لا تضف مكتبة لأجل وظيفة بسيطة يمكن تنفيذها محليًا بسهولة.
لا تكثر من المكتبات التي تؤدي أدوارًا متقاربة.
أي مكتبة جديدة يجب أن تخدم حاجة حقيقية.
18) قواعد خاصة بالتحديثات والتعديلات

عند تحديث ميزة موجودة:

افهم السلوك الحالي قبل تغييره.
لا تكسر تدفق المستخدم الحالي إلا إذا طُلب ذلك.
احتفظ بالتوافق الخلفي ما أمكن.
عند refactor:
لا تغيّر السلوك
غيّر البنية فقط
عند bug fix:
أصلح السبب الجذري
لا تكتفِ بإخفاء العرض الخارجي للمشكلة
عند تحسين UI:
لا تضر بالأداء
لا تكسر الـ responsiveness
لا تغيّر التفاعل الأساسي بلا داع
19) قواعد خاصة بالإصدار Release Readiness

قبل اعتبار العمل منتهيًا، يجب التحقق من:

عدم وجود أخطاء build
عدم وجود تحذيرات مهمة مهملة
عدم وجود أكواد غير مستخدمة
عمل التدفقات الأساسية
سلامة auth
سلامة الـ API calls
استقرار الواجهة
توافق الألوان والخطوط والمسافات
سلامة النصوص والترجمة
عدم وجود placeholder data أو debug leftovers
20) تعريف الإنجاز Done Criteria

لا يعتبر العمل مكتملًا إلا إذا:

تم تنفيذ المطلوب بدقة
لم ينكسر شيء قائم
الكود منظم وقابل للصيانة
الواجهة متناسقة
الأداء مقبول
استهلاك الشبكة والسيرفر محسوب
الأخطاء معالجة
الاختبارات أو التحقق الأساسي تم
التعديل موثّق إذا كان مؤثرًا
21) ممنوعات

يُمنع على الوكيل الذكي:

كتابة حلول مؤقتة غير موثقة وكأنها نهائية
نسخ كود متكرر بدون سبب
وضع منطق API في الواجهة مباشرة
تجاهل الأخطاء أو إسكاتها بلا logging
تنفيذ طلبات شبكية مكررة بلا حاجة
كسر التوافق مع المشروع الحالي بدون تنبيه واضح
استخدام حزم كثيرة بلا داع
إضافة debug code ونسيانه
كتابة UI غير متناسق مع بقية التطبيق
تحميل بيانات أو صور أكبر من الحاجة
إهمال RTL إذا كان التطبيق عربيًا
إهمال الأمان أو التحقق من الإدخال
22) أسلوب العمل المطلوب من الوكيل

عند كل مهمة:

افهم المطلوب بدقة
حدّد الملفات المتأثرة
نفّذ أقل تعديل صحيح يحقق الهدف
حافظ على البنية والنمط الحالي
حسّن الجودة عند الحاجة بدون مبالغة
راجع الأداء والأمان
تأكد من عدم كسر التطبيق
وثّق أي قرار مؤثر
23) معايير خاصة إذا كان التطبيق عربيًا
دعم RTL بالكامل
محاذاة صحيحة للعناصر والنصوص
عدم انعكاس الأيقونات الخاطئ
دعم الأرقام العربية أو الإنجليزية حسب قرار المنتج
مراعاة طول النصوص العربية
اختبار التنسيق على شاشات مختلفة
الحفاظ على وضوح الخط والقراءة
24) معايير خاصة إذا كان التطبيق SaaS أو متصل بخادم
تقليل عدد الطلبات
بناء طبقة cache واضحة
دعم pagination وfilters وsearch بشكل اقتصادي
عدم جلب بيانات dashboard الثقيلة دفعة واحدة
فصل الـ endpoints حسب الحاجة
دعم retry محدود ومدروس
تسجيل حالات الفشل
احترام timeouts
دعم session/token refresh بشكل آمن
عدم حجز موارد السيرفر بطلبات مستمرة غير ضرورية
25) الصيغة المطلوبة في أي مخرجات برمجية

عند تعديل أو إنشاء كود، يجب أن يكون الناتج:

منظمًا
كاملًا
جاهزًا للنسخ
لا يحتوي على أجزاء ناقصة غير مفسرة
يراعي الأمان
يراعي الأداء
يراعي الصيانة
يراعي تجربة المستخدم
يراعي أحدث الممارسات المناسبة لـ Flutter 2026
